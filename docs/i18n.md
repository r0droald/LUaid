# Internationalization (i18n)

Kapwa Help supports three languages: English (`en`), Filipino (`fil`), and Ilocano (`ilo`). Translations are loaded at runtime by `i18next-http-backend` from `public/locales/`.

## File Structure

```
public/locales/
  en/translation.json    ← Source of truth (English)
  fil/translation.json   ← Filipino (Tagalog)
  ilo/translation.json   ← Ilocano
```

Each file uses a two-level nested structure: `Namespace.key`. Current namespaces:

- `App`, `Navigation` — shell + cross-cutting UI
- `ReliefMap`, `PinDetail`, `HubDetail`, `HazardDetail` — map page
- `Dashboard`, `Transparency`, `Stories` — dashboard page
- `ReportForm`, `SubmitForm`, `DonationForm`, `PurchaseForm`, `HazardForm`, `ClaimForm` — report flows
- `Deployments`, `ReliefOps` — operations views

Namespaces are not enforced in the code — they're a convention in the JSON keys. Add a new namespace when introducing a new feature area; keep cross-cutting UI in `App` or `Navigation`.

## How It Works

- **Detection:** Language is detected from the URL path (`/:locale/...`), configured in `src/i18n.ts`
- **Fallback:** If a key is missing in the target language, `i18next` falls back to the English value
- **Interpolation:** Dynamic values use `{{variable}}` syntax (e.g., `"{{count}} organizations"`)
- **In components:** Use the `useTranslation` hook — `const { t } = useTranslation(); t("Dashboard.hero")`

For the agent-scoped version of this doc (auto-loads when editing `public/locales/`), see `.claude/rules/i18n.md`.

## Adding New Translation Keys

1. Add the key to `public/locales/en/translation.json` (English is the source of truth)
2. Run the translation script to machine-translate new keys into Filipino and Ilocano:

```bash
npm run translate
```

3. The script is **incremental** — it only translates keys that are missing or still in English. Human-reviewed translations are preserved.
4. Upload the updated `fil/` and `ilo/` files to Crowdin for human review

## Translation Script

The script at `scripts/translate.ts` uses `google-translate-api-x` (free, no API key) to machine-translate English strings.

### What it does

- Reads `en/translation.json` as the source
- For each target language, translates only **new or untranslated** keys
- Preserves `{{interpolation}}` variables by stripping them before translation and restoring after
- Writes output to `fil/translation.json` and `ilo/translation.json`

### Language code mapping

| Output folder | Google Translate code | Language |
|---------------|----------------------|----------|
| `fil/` | `tl` (Tagalog) | Filipino |
| `ilo/` | `ilo` | Ilocano |

### What gets skipped

- Keys where the existing translation already differs from English (already translated)
- Untranslatable values: brand names (`LUaid.org`), pure interpolation (`{{count}}`), URLs

### Example output

```
→ Translating to Filipino (tl → fil/)...
  · App.description — kept existing
  ✓ SubmitForm.title — Magsumite ng Ulat
  - App.title — LUaid.org (untranslatable)

  → Wrote public/locales/fil/translation.json (2 new, 65 kept)
```

## Workflow Summary

| Step | Action | Owner |
|------|--------|-------|
| 1 | Add/update keys in `en/translation.json` | Developer |
| 2 | Run `npm run translate` | Developer |
| 3 | Review machine translations on Crowdin | Native speakers |
| 4 | Pull approved translations from Crowdin | Developer |

Crowdin is the source of truth for reviewed translations. The script is a bootstrapping tool — it generates a first pass, not the final product.
